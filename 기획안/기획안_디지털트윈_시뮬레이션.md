# 도시 디지털 트윈 시뮬레이션 기획안

## 0. 현재 데이터로 가능한가?

### 결론: 가능하다. 단, "진짜 개인"이 아니라 "합성 개인(Synthetic Agent)"이다.

**현재 데이터의 실체:**
모든 SPH 데이터는 개인 단위가 아닌 **집계(aggregate) 데이터**다.
- 유동인구: "신당동 × 주중 × 남성 × 30대 × 오전 = 37.42명" (float)
- 카드매출: "신당동 × 개인 × 주중 × 여성 × 25세 × 점심 × 싱글 = 커피 10.27건" (float)
- 자산소득: "신당동 × 개인 × 남성 × 35세 = 314명, 평균소득 4,200만원" (integer)

**핵심 아이디어: 통계적 역분해(Statistical Decomposition)**

집계 데이터의 분포를 기반으로 **합성 개인(Synthetic Agent)**을 확률적으로 생성한다.
마치 인구조사 통계표에서 가상의 시민을 복원하는 것과 같다.

```
집계 데이터                    합성 개인
┌──────────────────┐          ┌──────────────────────────────────┐
│ 신당동, 남, 30대  │          │ Agent #4,271                     │
│ 거주인구: 37.42   │    →     │ 위치: 신당동 (거주지)              │
│ 직장인구: 12.08   │  확률적   │ 성별: 남, 나이: 32               │
│ 방문인구: 8.54    │  샘플링   │ 라이프스타일: L01 (싱글)           │
│                  │          │ 소득: 4,200만원, 신용: 1등급       │
│ 카드매출:         │          │ 직업: 일반직원, 주거: 20평대       │
│  커피 10.27건    │          │ 소비패턴: 커피 월12회, 음식 월22회  │
│  음식 22.31건    │          │ 이동패턴:                        │
│                  │          │   08시: 신당동(거주) → 여의도(직장) │
│ 자산소득:         │          │   12시: 여의도 주변 점심           │
│  평균소득 4,200   │          │   18시: 강남(방문) → 신당동(귀가)  │
│  대기업 비율 32%  │          │ 카페 방문 확률: 오전 78%, 오후 45% │
└──────────────────┘          └──────────────────────────────────┘
```

### 합성 개인 1명이 가질 수 있는 속성 (현재 데이터 기반)

| 속성 | 값 | 출처 |

## 1. 서비스 콘셉트

### "살아있는 도시" — 서울 디지털 트윈 시뮬레이터

지도 위에서 수만 개의 점(합성 시민)이 시간에 따라 이동하고, 소비하고, 생활한다.
사용자는 이 가상 도시 위에 **"만약에"를 던진다.**

- "여기에 카페를 차리면?"
- "이 동네에 정수기 렌탈 영업을 하면?"
- "새 아파트 단지가 들어서면 주변 상권이 어떻게 변할까?"

AI가 합성 시민들의 행동을 시뮬레이션하여 **예상 수익, 수요 변화, 최적 입지**를 답한다.

## 2. 합성 인구 생성 파이프라인

### Step 1: 앵커 인구 결정

자산소득 데이터의 `CUSTOMER_COUNT`가 유일한 정수값 → 이것을 기준 인구로 사용

```python
# 법정동별 × 성별 × 연령대별 인구수 (자산소득 기준)
# 예: 신당동, 남성, 30대 = 314명 → 314개의 에이전트 생성
anchor_population = asset_income.groupby(
    ['DISTRICT_CODE', 'GENDER', 'AGE_GROUP']
)['CUSTOMER_COUNT'].sum()
```

### Step 2: 속성 확률 분포 계산

각 (법정동 × 성별 × 연령대) 셀에 대해, 다른 데이터에서 조건부 확률 추출:

```python
# 라이프스타일 분포 (카드매출에서 추출)
# 신당동, 남성, 30대의 라이프스타일별 매출 비중 → 확률로 변환
lifestyle_prob = card_sales.query("district=='신당동' & gender=='M' & age==30") \
    .groupby('LIFESTYLE')['TOTAL_AMOUNT'].sum()
lifestyle_prob = lifestyle_prob / lifestyle_prob.sum()
# → {L01(싱글): 0.62, L02(신혼): 0.18, L03(영유아): 0.12, ...}

# 소득 분포 (자산소득에서 직접)
# 해당 셀의 소득구간별 비율이 이미 있음
income_prob = {
    '~2000만': 0.15, '2000~3000만': 0.22, '3000~4000만': 0.28,
    '4000~5000만': 0.18, '5000~6000만': 0.10, '6000~7000만': 0.05, '7000만~': 0.02
}

# 직업 분포 (자산소득에서 직접)
job_prob = {
    '대기업': 0.32, '일반직원': 0.28, '전문직': 0.15,
    '임원': 0.03, '일반자영업': 0.12, '전문자영업': 0.05, '기타': 0.05
}
```

### Step 3: 개인 에이전트 생성

```python
for district, gender, age_group in anchor_population.index:
    count = anchor_population[(district, gender, age_group)]
    
    for i in range(count):
        agent = SyntheticAgent(
            id=f"{district}_{gender}_{age_group}_{i}",
            home_district=district,
            gender=gender,
            age=random.randint(age_group, age_group + 4),  # 5세 구간 내 랜덤
            lifestyle=random.choices(lifestyles, weights=lifestyle_prob),
            income=random.choices(income_brackets, weights=income_prob),
            job_type=random.choices(job_types, weights=job_prob),
            housing_size=random.choices(housing_sizes, weights=housing_prob),
            credit_grade=random.choices([1,2,3], weights=credit_prob),
        )
        
        # 소비 패턴 할당 (20개 업종별 월 지출)
        agent.spending = assign_spending_pattern(district, gender, age_group, agent.lifestyle)
        
        # 이동 패턴 할당 (시간대별 위치)
        agent.movement = assign_movement_pattern(district, gender, age_group)
        
        agents.append(agent)
```

### Step 4: 이동 패턴 모델링

유동인구 데이터에서 **거주인구/직장인구/방문인구** 비율로 시간대별 위치를 결정:

```
시간대별 에이전트 위치 결정 로직:

T06 (06~09시, 아침):
  - 거주지에 있을 확률 = residential / (residential + working + visiting)
  - 직장으로 이동할 확률 = working / total
  - 기타 방문 확률 = visiting / total

T09 (09~12시, 오전):
  - 대부분 직장 위치 (working_population이 지배적인 법정동으로 이동)
  - 직장 법정동 결정: 해당 시간대 working_population이 높은 법정동에 확률적 배정

T12 (12~15시, 점심):
  - 직장 주변 법정동으로 단거리 이동 (방문인구 증가 패턴 반영)
  - 카드매출 음식/커피 비중이 높은 인접 법정동으로 분산

T18 (18~21시, 저녁):
  - 귀가 시작. 거주지 법정동으로 복귀 확률 증가
  - 일부는 유흥/문화 법정동으로 이동 (방문인구 패턴)

T21 (21~24시, 심야):
  - 대부분 거주지 복귀
```

**핵심**: 각 법정동의 시간대별 (거주/직장/방문) 인구 비율이 이미 데이터에 있으므로,
에이전트를 이 비율에 맞게 법정동 간 이동시키면 실제 유동인구 패턴과 일치하게 된다.

### PoC vs 프로덕션 기술 선택

| 레이어 | PoC (빠른 검증) | 프로덕션 (확장) |

| 지도 | Pydeck ScatterplotLayer | Deck.gl ScatterplotLayer (WebGL) |
| 애니메이션 | 시간대별 스냅샷 전환 | WebGL 실시간 파티클 애니메이션 |
| 백엔드 | Streamlit 내장 | FastAPI |
| DB | DuckDB (인메모리) | DuckDB + Redis (캐시) |
| AI | Claude API 직접 호출 | Claude API + 프롬프트 캐싱 |
| 합성인구 | 5만명 샘플 (3개 구) | 50만명+ (서울 전체) |

현재 데이터 기반 생성 가능한 에이전트 수:

| 지역 | 자산소득 CUSTOMER_COUNT 합계 (추정) | PoC 샘플링 |
|------|-----------------------------------|-----------|
| 중구 | ~35,000명 | 3,500 (10%) |
| 영등포구 | ~120,000명 | 12,000 (10%) |
| 서초구 | ~150,000명 | 15,000 (10%) |
| **합계** | **~305,000명** | **30,500명** |

PoC에서는 10% 샘플링으로 ~3만 에이전트를 생성.
Pydeck ScatterplotLayer는 10만 포인트까지 렌더링 성능 문제 없음.

## 7. 시뮬레이션 정확도와 한계

### 할 수 있는 것 ✅
| 시뮬레이션 | 정확도 | 근거 |
|-----------|--------|------|
| 시간대별 유동인구 분포 | ★★★★ | SPH 유동인구 데이터 직접 사용 |
| 업종별 소비 추정 | ★★★★ | 카드매출 20개 업종 세분화 |
| 인구 프로파일링 | ★★★★ | 성별×연령×소득×직업×라이프스타일 |
| 지역별 상대 비교 | ★★★★ | 동일 기준으로 법정동 간 비교 |
| 카페/음식점 매출 추정 | ★★★☆ | 커피/음식 카드매출 + 유동인구 |
| 이사 수요 예측 | ★★☆☆ | 전세 시세 변동 = 간접 지표 |

### 할 수 없는 것 (데이터 한계) ❌
| 항목 | 이유 | 보완 방법 |
|------|------|----------|
| 개인 단위 추적 | 집계 데이터, 개인 식별 불가 | 확률적 합성으로 대체 |
| 실시간 이동 | 시간대별 스냅샷만 존재 (7개 슬롯) | 시간대 간 보간(interpolation) |
| 경쟁 업체 반영 | 기존 카페/상점 위치 데이터 없음 | 카드매출 건수로 간접 추정 |
| 법정동 내부 위치 | 법정동 중심점만 가능, 건물 단위 X | GeoJSON 폴리곤 내 랜덤 배치 |
| 서울 3개 구 외 지역 | 리치고 데이터 범위 한정 | SPH만으로도 서울 전체 가능 |
| 개인 간 상호작용 | 입소문, 네트워크 효과 모델링 불가 | 향후 SNS 데이터 연동 시 가능 |

## 9. PoC 구현 계획

### Phase 1: 합성 인구 엔진 (Day 1~3)
- [ ] CSV → DuckDB 적재 (파일명 오류 매핑 포함)
- [ ] 자산소득 CUSTOMER_COUNT 기반 앵커 인구 생성
- [ ] 속성 확률 분포 계산 (라이프스타일, 소득, 직업)
- [ ] 합성 에이전트 ~3만명 생성 및 저장
- [ ] 시간대별 이동 패턴 할당

### Phase 2: 지도 시각화 (Day 3~5)
- [ ] GeoJSON 법정동 경계 추출 및 파싱
- [ ] Pydeck ScatterplotLayer 에이전트 렌더링
- [ ] 타임라인 슬라이더 (7개 시간대 전환)
- [ ] 색상: 라이프스타일별 구분
- [ ] 클릭 시 에이전트 프로파일 팝업

### Phase 3: 시뮬레이션 엔진 (Day 5~7)
- [ ] 카페/음식점 매출 시뮬레이션 로직
- [ ] 렌탈 수요 시뮬레이션 로직
- [ ] 수요 예측 모델 (시세 변동 + 유동인구 트렌드)
- [ ] 핀 드롭 → 시뮬레이션 결과 UI

### Phase 4: AI 에이전트 연동 (Day 7~9)
- [ ] Claude API 연동
- [ ] 시뮬레이션 결과 해석 프롬프트
- [ ] 대화형 시나리오 질의 ("규모를 늘리면?", "옆 동네로 옮기면?")
- [ ] 1번 기획안 통합 (탭 구조)

### Phase 5: 데모 준비 (Day 9~10)
- [ ] 시나리오 3개 준비 (카페, 렌탈 영업, 수요 예측)
- [ ] 성능 최적화 (에이전트 수 조절)
- [ ] 데모 스크립트 작성
