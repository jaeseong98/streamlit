# 동네 엑스레이 (Neighborhood X-Ray) — 해커톤 상세 계획

## 1. 아이디어 개요

### 한 줄 요약
**"법정동 하나를 클릭하면, 그 동네의 돈·사람·집·서비스를 한 화면에 엑스레이처럼 보여주고, 아직 아무도 모르는 '넥스트 핫플'을 예측하는 지도 대시보드"**

### 풀고자 하는 문제
- 프랜차이즈 출점, 부동산 투자, 마케팅 타겟팅 시 **"이 동네가 좋은가?"**를 판단하려면 여러 사이트를 뒤져야 함
- 유동인구, 소비패턴, 부동산 시세, 인구구조, 서비스 수요 — 모두 따로 놀고 있음
- 특히 **"앞으로 뜰 동네"**를 정량적으로 예측하는 도구는 시장에 없음

### 데이터 퓨전의 핵심 가치
| 데이터 소스 | 알 수 있는 것 |
|------------|--------------|
| SPH | 이 동네 사람들이 **누구이고**, **얼마를 벌고**, **어디에 쓰는지** |
| 리치고 | 이 동네의 **부동산 가치**와 **인구 구조 변화** |
| 아정당 | 고객이 **어떻게 유입되고**, **뭘 사고**, **얼마나 전환되는지** (풀퍼널) |

→ 3개를 결합하면 **단일 데이터로는 절대 볼 수 없는 입체적 동네 인사이트**가 나옴

### 타겟 사용자
1. **부동산 투자자**: 어디 사야 하나? 어떤 동네가 오를까?
2. **프랜차이즈/소매 브랜드**: 어디에 매장을 열어야 하나?
3. **마케팅 담당자 (아정당 등)**: 어떤 동네에 광고비를 집중할까?
4. **지자체 도시계획**: 어떤 동네에 인프라 투자가 필요한가?

## 2. 핵심 기능 (4개 탭)

### 탭 1: 동네 지도 (메인)
- 서울 법정동 단위 **코로플레스 맵** (GeoJSON 경계선)
- 색상 = 선택한 지표 (소비력/유동인구/부동산시세/종합점수)
- 마우스 호버 → 동네명 + 핵심 수치 툴팁
- 클릭 → 탭 2로 이동 (해당 동네 상세)

### 탭 2: 동네 프로파일 (상세)
법정동 하나를 선택하면 4개 섹션으로 엑스레이:

**A. 소비 DNA** (SPH 카드매출)
- 20개 업종별 매출 비중 → **레이더 차트**
- "이 동네는 커피·미용에 강하고, 주유·자동차에 약하다" 한눈에 파악
- 주중 vs 주말, 시간대별 소비 패턴

**B. 사람 흐름** (SPH 유동인구)
- 시간대별 거주/근무/방문 인구 → **영역 차트**
- "이 동네는 낮에 사람이 빠지고, 저녁에 방문객이 몰린다" 패턴 식별
- 성별/연령별 인구 구성 → **인구 피라미드**

**C. 부동산 트렌드** (리치고)
- 매매가/전세가 12년 추이 → **라인 차트** (공급평당 만원)
- 매매-전세 갭 변화 → 투자 수요 지표
- 세대수, 영유아 비율 등 인구 구조

**D. 금융 건전성** (SPH 자산소득)
- 평균소득, 중위소득, 소득 분포 → **히스토그램**
- 신용점수 분포, 연체율
- 직업군 비율 → **도넛 차트**

### 탭 3: 넥스트 핫플 예측
**5개 선행지표를 조합한 "핫플 스코어":**

| # | 지표 | 데이터 소스 | 의미 |
|---|------|-----------|------|
| 1 | 방문인구 증가율 | SPH 유동인구 (VISITING_POPULATION) | 아직 안 살지만 놀러오기 시작 |
| 2 | 카페·식음료 매출 증가율 | SPH 카드매출 (COFFEE_SALES + FOOD_SALES) | 상권이 형성 중 |
| 3 | 20~30대 유입 비율 변화 | SPH 유동인구 (AGE_GROUP 20~35) | 트렌드 세대 이동 |
| 4 | 매매가 상승률 | 리치고 (MEME_PRICE_PER_SUPPLY_PYEONG) | 투자 수요 유입 |
| 5 | 인터넷 신규 설치 증가율 | 아정당 V05 (OPEN_COUNT) | 실제 전입 발생 |

- 각 지표를 0~100으로 정규화 → 가중 평균 → **핫플 스코어 (0~100)**
- 지도에 히트맵 오버레이
- **백테스트**: 2022~2023 데이터로 예측 → 2024 실제 가격 상승과 비교 → 정확도 시연

### 탭 4: 동네 비교
- 2~3개 동네 나란히 비교
- 모든 지표를 테이블 + 차트로 병렬 표시
- "A동네 vs B동네, 어디가 더 유망한가?"

## 4. 데이터 전처리 전략

원본 CSV가 크기 때문에 (카드매출 1GB, 유동인구 131MB) **1회 전처리 후 경량 parquet로 저장**, 앱에서는 전처리 데이터만 로딩.

### Step 1: GeoJSON 추출
- **소스**: `SPH_카드매출.csv` (실제 내용: 법정동 마스터, GeoJSON 경계 포함)
- DISTRICT_GEOM 컬럼 파싱 (과학적 표기법 좌표 → 일반 float 변환)
- 법정동 코드 → GeoJSON Feature 매핑
- **출력**: `geo_boundaries.json` (FeatureCollection)

### Step 2: 카드매출 집계
- **소스**: `SPH_코드_마스터.csv` (실제 내용: 카드매출, 6.2M행, 1GB)
- 집계 A: 법정동 × 년월별 20개 업종 매출 합계 (성별/연령/시간 모두 합산)
- 집계 B: 법정동 × 최근 년월 × 업종별 상세 (시간대, 주중/주말 breakdown 유지)
- **chunked read** (10만행씩) → 메모리 절약
- **출력**: `card_sales_agg.parquet`

### Step 3: 유동인구 집계
- **소스**: `SPH_유동인구.csv` (2.6M행, 131MB)
- 집계 A: 법정동 × 년월별 거주/근무/방문 합계
- 집계 B: 법정동 × 최근 년월 × 시간대별 상세 (연령/성별 breakdown)
- **출력**: `population_agg.parquet`

### Step 4: 자산소득 집계
- **소스**: `SPH_법정동_마스터.csv` (실제 내용: 자산소득, 269K행)
- 법정동 × 년월별 가중 평균 (소득, 신용점수, 연체율, 직업군 비율)
- 가중치 = CUSTOMER_COUNT
- **출력**: `income_agg.parquet`

### Step 5: 부동산/인구
- 리치고 데이터 소량 (4,357행, 118행) → 그대로 parquet 변환
- **출력**: `realestate.parquet`

### Step 6: 아정당 집계
- V05 (8,380행) 시군구 단위 → 그대로 parquet 변환
- **출력**: `install_agg.parquet`

## 5. 데이터 조인 전략

3개 데이터소스의 지역 코드 체계가 다른 것이 핵심 과제:

| 데이터 | 지역 키 | 단위 |
|--------|---------|------|
| SPH | PROVINCE_CODE + CITY_CODE + DISTRICT_CODE | 법정동 |
| 리치고 | BJD_CODE (10자리) | 읍면동/시군구 |
| 아정당 | INSTALL_STATE + INSTALL_CITY (텍스트) | 시군구 |

### 조인 방법
1. **SPH 법정동 마스터** (`SPH_카드매출.csv`)를 **브릿지 테이블**로 활용
   - PROVINCE_CODE/CITY_CODE/DISTRICT_CODE ↔ PROVINCE_KOR_NAME/CITY_KOR_NAME/DISTRICT_KOR_NAME
2. **리치고 ↔ SPH**: BJD_CODE ↔ DISTRICT_CODE 코드 매핑 테이블 생성
3. **아정당 ↔ SPH**: INSTALL_STATE/CITY ↔ PROVINCE_KOR_NAME/CITY_KOR_NAME 텍스트 매칭
4. 아정당은 시군구 단위이므로, 지도에서는 **시군구 집계 레벨**로 표시

### 데이터 커버리지 제약
- **리치고**: 서울 중구/영등포구/서초구 3개 구만 커버
- **아정당**: 전국 18개 시도 커버 (시군구 단위)
- **SPH**: 전국 법정동 단위 커버
- → 지도에 "리치고 데이터 가용 여부" 표시, 없는 구는 SPH 데이터만 표시

## 6. 핫플 스코어 알고리즘

### 수식
```
핫플 스코어 = w1 × 방문인구_증가율_정규화
            + w2 × 카페식음료_매출_증가율_정규화
            + w3 × 2030_유입비율_변화_정규화
            + w4 × 매매가_상승률_정규화
            + w5 × 신규설치_증가율_정규화
```

### 파라미터
- **증가율 계산**: 최근 6개월 평균 vs 이전 6개월 평균
- **정규화**: Min-Max 정규화 (0~100)
- **가중치 기본값**: w1=0.25, w2=0.20, w3=0.20, w4=0.20, w5=0.15
- → UI에서 슬라이더로 사용자 조정 가능

### 백테스트 로직
1. 2022년 H2 데이터로 핫플 스코어 산출
2. 2023~2024 실제 매매가 상승률과 상관관계 측정 (피어슨 상관계수)
3. 상위 10% 스코어 동네 vs 하위 10% 동네의 실제 가격 변화 비교
4. 결과를 차트로 시각화 → 발표 시 신뢰도 근거

## 8. 기술 리스크 & 대응

| 리스크 | 대응 |
|--------|------|
| GeoJSON 파싱 실패 (이스케이프, 과학적 표기법) | json.loads 전에 문자열 정리, ast.literal_eval 폴백 |
| 카드매출 1GB 메모리 부족 | chunked read (10만행)로 집계 후 parquet 저장, 원본 다시 안 읽음 |
| 법정동 코드 매핑 불일치 (SPH ↔ 리치고) | 코드 매핑 실패 시 한글명 기반 fuzzy matching 시도 |
| 리치고 데이터가 서울 3개 구만 커버 | 지도에 "리치고 데이터 가용" 명시 표시, 없는 구는 SPH만 |
| Streamlit + Folium 클릭 이벤트 한계 | streamlit-folium의 `st_folium(returned_objects=["last_object_clicked"])` 활용 |
